## Project Overview
GregSweeper — modern Minesweeper game built with vanilla JavaScript (ES6 modules). No framework, no bundler, no package.json. Deployed on GitHub Pages.

## Development

**Local server:** `python -m http.server 8080` (or use `GregSweeper.bat` which opens the browser too).
No build step, no npm install, no dependencies to manage. Firebase SDK loaded via CDN.

**Testing:** Pure-logic regression suite in `test/` (Node's built-in `node --test`, no dependencies) — solver move-type invariant, reveal gating (`test/revealGating.test.mjs`, with real canonical-board fixtures in `test/fixtures/`), gauss soundness (`test/gaussSoundness.test.mjs`), bomb info-value pricing, par/serialize round-trip, name filter, streak, chord, dates/cache, rules contract, power-up sync, resume eligibility (`test/resumeEligibility.test.mjs` — stale daily/weekly saves, date-rollover expiry), score-row matching (`test/scoreRowMatch.test.mjs` — cross-device completion adoption + submission dedupe). Runs in CI via `.github/workflows/test.yml` on every push/PR to main/test; keep it green before merging. For UI and modifier verification, use Playwright via MCP — load the live page with `?seed=<custom>` to play deterministic boards.

**Deploy:** Push to `main` branch triggers GitHub Pages (via the `.github/workflows/deploy-pages.yml` Actions workflow — composes main at `/` and the `test` branch at `/test/`). After push, bump `CACHE_NAME` in `sw.js` (format: `gregsweeper-v1.6.N`). The `test` branch deploys to `https://christopherwells.github.io/GregSweeper/test/` with an orange "TEST BUILD" banner and skips Firebase writes via `isTestEnvironment()`.

## Modifier Load-Bearing Filter (generation)

Beyond solvability, every generation path (precompute, daily play-path local-gen, weekly play-path local-gen, challenge) requires that each non-mystery modifier on a board contributes at least one deduction the player needs to make. Implemented via `findDecorativeGimmicks` in `boardSolver.js`: strip one gimmick type's info at a time and re-run the solver — if the board is still solvable, that type is decorative on this layout and the candidate is rejected.

- **Mystery is exempt** — it removes info by definition, can't be load-bearing. Every mystery board would fail the test, so we don't apply it.
- **Walls and locked are exempt** — they're structural (topology change, reveal-order change), not deduction contributors. Stripping them would invalidate the cell numbers.
- **Sonar / compass / wormhole / liar / mirror** are testable. The filter strips their constraints (and for mirror, the un-swap assumption) and asks "could the player still solve this board with only the other modifiers' info, AT THE SAME DIFFICULTY?"
- **Per-type, not per-cell.** A board with 3 sonar cells where only 1 contributes passes — sonar AS A TYPE contributed. We don't iterate per-cell; that's intentional.
- **A modifier "contributes" when stripping it does ANY of**: makes board unsolvable, raises technique level (Pass C tank/gauss instead of Pass A), or adds ≥2 clicks. The threshold catches both strict load-bearing AND meaningful shortcut — a sonar that lets the player skip a tank-solve counts even if Pass C could have eventually solved without it. See `_gimmickContributes` in `boardSolver.js`. The 2-click threshold avoids penalizing incidental savings (1 saved click is often just "the same cell deduced moments later via a neighbor").
- **Fallback when no candidate contributes**: precompute falls back to the highest-scoring solvable candidate (with a console note); play-path retry loops relax the requirement after RELAX_AFTER_BASES (challenge) / LOAD_BEARING_BUDGET=25 (daily/weekly) attempts. Better to ship a decorative-modifier board than spin forever.
- **Why this matters**: an audit (2026-05-10) found ~3-5 of the prior 13 dailies shipped decorative modifiers (sonar, compass, or liar that contributed nothing). With force-injected coverage missions pushing under-sampled gimmicks, the natural rate of "decorative" was rising. This filter makes the modifier MEAN SOMETHING on the board — either required, or at least a useful shortcut.

## The No-Guess Contract (status per 2026-06-10 adversarial audit)

**Where it holds, and how:** `generateBoard` is NOT self-certifying — its terminal fallback returns the best-effort (possibly unsolvable) board, so the contract lives in caller-side verification. Challenge/timed: certified from the player's ACTUAL first click; the relax tiers (technique floor, load-bearing) never relax solvability — worst case regenerates forever, never ships uncertified. Daily/weekly: certified from the board CENTER at precompute + every load; the rendered "Start here" cell is a separately-searched certified full-solve anchor. Hard gates added 2026-06-10: precompute/regenerate scripts THROW rather than write an uncertified canonical; the client strip-modifiers fallbacks verify (bounded retry) before shipping/writing; first-click mine relocation was REMOVED 2026-06-12 (frozen board: a first-click mine is a bomb hit, the canonical never mutates); the post-tutorial warm-up board uses the certified generator. Daily/weekly bomb strikes are information-monotone (mine preserved, numbers unchanged, strike = free flag) — they can never create a guess state. A revealed mine is modeled as a KNOWN MINE (sim=2) in `findDeducibleFrontier` — modeling it as revealed-safe poisoned every adjacent constraint after a strike (fixed 2026-06-10; regression-tested).

**The per-state honesty asterisk:** the certificate guarantees a no-guess PATH from the certified anchor, not that every reachable state is deduction-complete. A daily player who skips the marked start can legitimately reach proof-free states (measured: ~71% of off-path first clicks) — the board stays winnable via the start cell, and the Lens points back to it instead of false-alarming. Chaos is OUTSIDE the contract by design and is now labeled so in the UI (mode card "No guarantees", help-modal line); site metadata says "every standard board".

**Reveal gating (closed 2026-06-10, per-board):** sonar / compass / wormhole constraints are usable by the solver only once their number is on screen — the origin cell revealed; for wormhole, EITHER endpoint (both display the pair sum). Implemented as `gimmickConstraintVisible` in `boardSolver.js`, applied in `isBoardSolvable` (Pass A push + `buildGimmickRuntimeConstraints` for Pass B/C) and `findDeducibleFrontier`. The gate's default comes from the BOARD's own contract flag: `createEmptyBoard` stamps `board._gatedCert = true`, `serializeBoard`/`deserializeBoard` carry it as `gatedCert` in canonical payloads (rules-validated boolean), and `gamePersistence` saves/restores it — so newly generated boards certify gated on every solver surface (generation acceptance, load re-certification, features, start-cell search, lens, receipts, bomb pricing) while historical canonicals (no flag) keep their ungated contract verbatim. `options.gateGimmickOrigins` overrides for tooling. Cross-contract asymmetry: gating only removes constraints, so a gated certificate ALMOST always implies an ungated one — not a theorem, because a wide ungated gimmick constraint can merge components past the 20-unknown tank limit into the (sound but incomplete) gauss path, occasionally proving LESS with more information (~3% of probe boards). Harmless: fetched canonicals are used VERBATIM — the client's load-time solve only feeds features / par / start-cell, it never rejects the board. Measured before shipping (`scripts/measure-reveal-gating.mjs`, 61 canonicals): 41 unaffected, 16 identical, 2 feature-drift (already played), 2 lose certification — weekly 2026-05-04 (past, left as history) and daily 2026-06-14 (precomputed ahead; regenerate before it goes live). Feature determinism: both precompute scripts now also write `dailyMeta/{date}` (and `dailyMeta/{weekStart}_weekly_first`) at generation time, so the generator's gated counts land before any client — possibly on older code, computing ungated counts — can upsert its own. Residual accepted risk: a client that cannot reach Firebase at all on a gimmick-board date locally generates with gated acceptance and can diverge from an old-contract canonical (same class as any code-version drift; the canonical-board architecture absorbs it for everyone online). Same commit also fixed a related certifier hole: liar stacks freely on base-value gimmicks, and `buildStaticGimmickConstraints` used to emit the liar-adjusted `displayedMines` as an EXACT constraint — a false premise that could certify a safe cell as a provable mine on challenge L81+ liar+sonar/compass/wormhole stacks. Such cells now contribute nothing, matching `isPureLiar`'s contract (zero canonical boards ever carried one; regression-tested in `test/revealGating.test.mjs`).

**Known remaining gaps (documented, not yet fixed):**
- **magnetPull: RESOLVED 2026-06-11** — redesigned from relocation to EXTRACTION (mines in the 3×3 are removed from the board entirely, revealed as defused markers, `state.totalMines` decremented). Removal is information-monotone — numbers only drop, no mine ever lands on a proven-safe cell — so the certificate survives and the liar-clamp exploit class is gone. Pinned by `test/magnetExtract.test.mjs`.
- **Locked-cell certification oracle: RESOLVED 2026-06-12** — the certifier no longer auto-reveals freed locked cells via ground truth; an unlocked cell is an ordinary unknown the passes must PROVE safe (still-locked cells are excluded from deduce-reveal sites so counters can't spin), and zero-valued cells now act as Pass A constraint sources (a zero adjacent to a locked cell is a real proof the old flood-exclusion missed). `test/lockedOracle.test.mjs` pins the first constructed counterexample (a 50/50 behind a lock certified under the oracle, refuses now). All shipped locked canonicals (dailies 2026-06-14/16, weeklies 2026-06-08/15) re-certify under the honest solver — no regenerations were needed.
- **Pressure plates: MITIGATED 2026-06-12 (calibration, not proof)** — the countdown is now par-calibrated via `plateSeconds` in difficulty.js: Pass-A-resolvable disarm work keeps the classic 10s/step rate, and each target the Pass-A estimator could NOT resolve (needs subset/tank reasoning — previously nearly free, the systematic under-timing) is billed at PLATE_TIER_WEIGHT × the par model's dearer tier coefficient, with a never-below-classic per-target floor, PLATE_MIN_SECONDS, and a PLATE_MAX_SECONDS cap. The certificate still doesn't model wall-clock (an unusually slow player can lose to a deadline) — the deadline just no longer ignores reasoning difficulty, and it rescales automatically with every refit. Pinned by `test/plateSeconds.test.mjs`.
- **First-click mine relocation: RESOLVED 2026-06-12 — the board is FROZEN.** Daily/weekly first-click mines are no longer relocated; the click falls through to the standard bomb-hit path (info-value penalty, strike marker, no game over) like any other mine click. Zero divergence: every player's canonical board is byte-identical for the whole date, and submitted features always describe the played layout. The marked start cell remains the certified safe entry; ignoring it is the player's choice.

## Board Solver Architecture

Two-layer solver guarantees no 50/50 guesses:

1. **Board solver** (`boardSolver.js`): Simulates full game playthrough from first click. Wall-aware neighbor lists. Gimmick handling via `getPlayerVisibleCount`:
   - **Mystery / sonar / compass / wormhole:** return UNKNOWN (no direct per-cell constraint; sonar/compass/wormhole contribute separate gimmick-range constraints in Pass A/B/C — reveal-gated on `board._gatedCert` boards, see the No-Guess Contract section). Liar-stacked gimmick cells contribute NO range constraint (their displayed value includes the lie).
   - **Liar:** return UNKNOWN. Liar's displayed value is true count ±1 — a disjunctive constraint, not a single value. `buildLiarConstraints` emits `{allowedMines: [X-1, X+1]}` for Pass C tank solver.
   - **Mirror:** return `cell.adjacentMines` (the true count). The displayed value shows the partner's count for visual deception; a smart player who recognizes the pair can mentally un-swap. Using `displayedMines` would create false constraints that made post-gimmick boards spuriously unsolvable.
   - Returns `{ solvable, remainingUnknowns, totalClicks, techniqueLevel }` where `techniqueLevel` is 0 (Pass A), 1 (Pass B subset), 2 (Pass C tank/gauss), or 3 (required disjunctive liar reasoning).

2. **Constraint solver** (`constraintSolver.js`): Union-find partitions independent constraint groups. Constraints use `allowedMines: number[]` — single-element for exact, multi-element for disjunctive (liar) — plus an optional `origin` (the source cell index) for provenance. Tank solver (bitmask brute-force with disjunctive check) for <=20 unknowns; Gaussian elimination for larger groups (exact constraints only — disjunctive filtered out before gauss). Gauss is sound-or-skip (reworked 2026-06-10): float row-reduction with partial pivoting, then a row yields deductions ONLY if every coefficient and the RHS snap to integers within 1e-6, extracted via the binary-variable bound rule (rhs == min/max achievable). The old extraction `Math.round`ed coefficients before pattern-matching — 0/1 systems legitimately produce 0.5 entries under elimination, and on unsatisfiable systems (wrong player flag) it fabricated "provable mine" verdicts (`test/gaussSoundness.test.mjs` pins both the fabrication case and a witness-consistency sweep). Gauss remains INCOMPLETE by design — a missed deduction is honest, a false one never is. `solveConstraints` returns, besides the flat `mines`/`safe` sets: `groups` (per-component `{unknowns, hasDisjunctive, origins}`), `cellGroup` (cell → group index for every solved cell), and `contradiction` (a tank component with zero satisfying assignments — on live player state this means a provably wrong flag).

**Deduction substrate (2026-06-10)** — the engine-as-witness layer the receipts/lens surfaces consume:
- **Trace**: `isBoardSolvable(..., { trace: true })` returns `result.trace = [{cell, tier, sources}]` — one entry per deduced reveal, where `sources` are the origin cells of the constraints that PROVED it (Pass A: the one constraint; Pass B: the subset pair; Pass C: the whole union-find component — the honest minimal explanation for enumeration). Collection-only: counters, ordering, and behavior are unchanged; generation hot loops don't pass the flag. Invariant: `trace.length + 1 === totalClicks`.
- **Per-group disjunctive attribution**: a Pass C deduction counts as `disjunctiveMoves` (and promotes techniqueLevel to 3) only if ITS union-find component carries a liar constraint. The old version batch-flagged every Pass C deduction of a round whenever any liar constraint existed anywhere, inflating `disjunctiveMoves` in dailyMeta on liar boards and over-promoting techniqueLevel. Consequence: on liar boards, `advancedLogicMoves` (the model's searchMoves) can be higher and `disjunctiveMoves` lower than under the old counting — the honest direction; historical dailyMeta rows written by old clients carry the old counting.
- **`findDeducibleFrontier(board, {respectFlags})`**: everything provable from the live state — full safe + mine frontier, each with `{tier, sources}`, plus `contradiction`. `respectFlags: false` runs flags-blind: player flags are CLAIMS, not facts, and one wrong flag can poison the constraint system into certifying a mine "provably safe" — every player-facing verdict must use the flags-blind run. `findNextSafeMove` is now a thin first-safe-cell wrapper over it.
- **`detectWrongFlags(board)`**: dual-solve diff — a player flag is provably wrong when the flags-blind run proves that cell safe; the flags-respecting run's `contradiction` additionally says "some flag is wrong" without localizing it.
- **Click timeline**: `state.clickTimeline = [{t, r, c, a}]` (a = r/f/u/c) records every player action via `recordPlayerAction` in gameState.js — reset per game, persisted with the auto-save, capped at 2000. The ground truth for player-grounded receipt claims (never narrate the solver's order as the player's).

**Player-facing proof surfaces (2026-06-10)** built on the substrate:
- **The Receipt** (`src/ui/receiptRenderer.js` + winLossHandler): post-loss, the full flags-blind frontier is painted (`.frontier-safe` outlines) with an honest verdict count, and tapping any cell in the explore view answers "was this knowable?" with a pulse of its proving region. Daily/weekly bomb hits get a one-line strike verdict (provable mine / safe cells existed / genuinely at the frontier) computed from the pre-strike state. Daily/weekly wins get a one-line board's-confession receipt: the crux (first non-trivial trace deduction) + the modifier's CERTIFIED contribution tier from `gradeGimmickContribution` (required / technique / shortcut / decorative — the copy can never overclaim). Suppressed for newcomer dailies (≤3 plays), same gate as the par breakdown.
- **The Lens** (the `#stuck-btn` "Stuck?" pill beside the flag toggle → `handleLensRequest` in receiptRenderer): Socratic mid-game help that never names the safe cell. Priority: (1) "one of your flags is provably wrong" (dual-solve diff, deliberately not localized); (2) pulse the proving region of the next available deduction, copy sized honestly by tier (a whole-component pulse SAYS it needs enumeration); (3) an empty frontier is impossible by construction on official boards — if seen, it's reported via reportCaughtError. Every invocation is recorded to `state.hintEvents` and submitted with daily scores; the refit excludes hinted plays. Mid-game responses render on the **board coach line** (`#board-coach` under `#board-container`, `src/ui/boardCoach.js`, shared styling with the Gym's `.lexicon-coach`) — persistent and re-readable, updated in place, cleared by the next reveal / flag / chord (`clearBoardCoach` at the three `recordPlayerAction` sites and in `newGame`). Only the pre-game "lens works mid-game" guard still toasts.
- **The Certificate stamp (2026-06-12, the front-door PR)**: `state.boardCertificate = { clicks, tier }` built by `certificateFromCheck` in boardSolver.js (null unless the check certified a full clear — a failed solve must never stamp). Stamp sites: daily and weekly use the bestStart cell's OWN full-solve check (the chip's claim is "solvable without guessing from the marked start", so the center check that feeds features/par is the wrong witness; no full-solve anchor → no stamp); challenge and timed use the accepted check from the generation loop (the contract runs from the player's actual first click); chaos never stamps (its modifiers apply post-loop unverified). Persisted/restored by gamePersistence. Rendered as the ✓ Certified chip in `#active-gimmick-bar` with a plain-language modal (`showCertificateModal` in headerRenderer.js: deduction count = clicks - 1, tier in plain words, daily/weekly add the generated-and-recheck line); chaos shows the inverse "No guarantees" chip on `#chaos-modifier-bar`, opening the same modal's exception copy. Companion front-door copy: title headline "No guesses. Ever." (title screen + in-game header) with the proof subline, the guarantee as the first Basics block in help, and the unnumbered tutorial promise step (`id: 'promise'`).
- **Reveal Safe power-up** is deduction-first: it reveals the next PROVABLY-safe cell (a worked example) and only falls back to the old random-safe oracle at a genuine frontier. X-Ray/Scan remain oracles but power-ups are structurally empty in daily/weekly/timed, so they never touch instrumented modes.
- **Greg's voice** (`src/logic/gregVoice.js`, pure + node-tested): the meta-layer made legible, with hard rules — Greg never says a number the engine can't prove, the bad days speak too, and each surface renders once per session. Three surfaces: (1) **Field Note** on the daily title card — derived from the CANONICAL BOARD via `fieldNoteFromBoard` (the stamped `missionTarget`/`missionIsPrimary` when present, else the board's actual `activeGimmicks`), NEVER by re-deriving the seed's slot against the current experimentTarget.json: boards are pre-generated up to 7 days ahead and the nightly refit reorders the coverage list, so slot re-derivation names the wrong gimmick (2026-06-10 incident: a wormhole board labeled a compass study); (2) **yesterday note** in the leaderboard modal — the closed loop from modelHistory.json diffs, with all four honesty branches first-class (tightened / WIDENED / fit-rejected / nobody-played); (3) **Lab File** — when handicaps.json v2 ships `handicapDetails` ({clean, bomb} per uid, emitted by both refit paths), the win modal's "Your par" becomes an itemization: `Your par 110.5s = Greg 96.0s your pace +8.4s bombs +6.1s`. No decomposition shipped → the plain line renders; the split is never fabricated client-side.
- **Greg’s Gym** (player-facing name; modules keep their original lexicon filenames) (`src/logic/lexicon.js` + `src/ui/lexiconUI.js`, title-screen 🏋️ button, lazy-loaded): generated single-technique lessons behind the **deducibility click-gate** — clicking a cell that is not currently provably safe bounces and pulses the proving region of an available deduction; the player physically cannot luck through, and the pattern is NAMED only after they perform it. Flags are gated the same way (right-click / long-press): a flag only sticks on a square the clues can settle as a MINE, so a flag is a worked deduction, never a guess marker; tapping a flagged square unflags it. The admission predicate IS the curriculum: `LESSONS.subset12` accepts only boards with `canonicalSubsetMoves >= 1` and nothing harder (no generic subsets / tank / disjunctive). Measured yield (`scripts/measure-lexicon-yield.mjs`): 1-in-3 to 1-in-5 at small sizes, <=8ms per accepted board — live generation, no pre-baked content. Tested invariant: every lesson is completable through provably-safe clicks alone (the gate can never soft-lock). The gym is fully voiced through the theme sound engine (reveal/cascade/flag/unflag/win + `playGateBounce`, a soft unthemed "not yet" for gate rejections) and supports CHORDING — gym flags are gate-proven mines, so a satisfied number has proven its remaining neighbors safe and the chord can never hit a mine (a half-flagged chord attempt gets the teaching bounce; an idle tap on an unflagged number stays silent). A mines-left counter + board number sit above the grid (the same anchor the main LCD provides). **In-the-moment coaching** (`_celebrate` + the `.lexicon-coach` line in lexiconUI.js): the gate teaches on failure (Socratic bounce), success notes teach on success — every tier-1 subset move is named from the two clue digits on screen ("Excellent use of the 1-2 pattern!"), the FIRST per board carries a recognition tip (overlap-subtraction framing from the classic pattern vocabulary: the 2's leftover square is a mine, the 1's leftover is safe, flagged mines reduce a 2-3 to a 1-2), later ones get short rotating affirmations; proven-mine flags get their `explainDeduction` reasoning once per board then a nod; first chord per board explains the mechanic; tier-0 safe reveals stay deliberately silent. ALL gym messages (bounces and cheers alike) render on the coach line UNDER the board, updated in place — global toasts queue serially, so under quick play a queued note would land seconds after the move it praises. The completing tier-1 move still gets its cheer (on these small boards the pattern move is often the LAST move; a blanket suppress-on-completion ate exactly the note the coaching exists for), with the naming paragraph carrying the full explanation. Prototype scope: one technique; expand only if it earns play.

**Generation (challenge mode, `gameActions.js`):** Retry loop with two layers of "smart":
- **Constructive mystery placement** — mystery cells are placed one at a time, verified after each; kept only if the board stays solvable. May place fewer than requested intensity. Other gimmicks (sonar/compass/wormhole/wall/liar/etc.) contribute info, so random placement works for them.
- **Technique-floor verification** — each level demands a minimum `techniqueLevel` via `getRequiredTechnique(level)`: L1-30→0, L31-60→1, L61+→2. Boards below the floor are rejected. After 15 base-board attempts without success, the floor relaxes to 0 so we don't spin forever.
- Up to 25 gimmick re-rolls per base board before regenerating the (expensive) base board. Daily strips gimmicks if unsolvable. Chaos is exempt from solvability.

## Important Patterns

- **DOM structure:** `#title-screen` is a sibling of `#app`, not inside it. Modals are inside `#app` — when `#app` has `.hidden` class (display: none), child modals can't render. Modals that need to show on title screen must be outside `#app`.
- **`_returnToTitle` flag** in main.js tracks when modals were opened from the title screen.
- **`$()` / `$$()`** are querySelector/querySelectorAll helpers from `src/ui/domHelpers.js`.
- **"Gimmicks" in code = "Modifiers" in player-facing UI.** Always use "Modifier" in user-facing text.
- **Daily mode** uses seeded RNG (`createDailyRNG(dateString)`) for deterministic boards per date.
- **`CURRENT_VERSION`** in main.js drives the "What's New" badge: first load after an update shows a NEW badge on the title-screen What's New button; the modal opens on click (it does NOT auto-open).
- **`generateBoard` calls `cleanSolverArtifacts`** on every return path (solver leaks `isRevealed` onto cells).

## Service Worker (sw.js)

Network-first with cache fallback. `ignoreSearch: true` on cache.match. Install uses `cache: 'reload'` to bypass HTTP cache. Activation deletes all old cache versions. Core assets pre-cached; non-default themes lazy-load.

**Cache busting:** Only bump `CACHE_NAME` in `sw.js`. No per-file `?v=` query strings on imports. SW registers on localhost too — unregister via DevTools or add `?v=N` when testing locally.

## Game Modes
- **Challenge (normal):** 120 levels, sawtooth difficulty, checkpoints every 5 levels, modifiers from L11+
- **Timed:** Race the clock, 4 difficulty tabs (Beginner/Intermediate/Expert/Extreme). Every timed board gets a par at generation (same PAR_MODEL as daily; gimmick terms zero on gimmick-free boards) shown as a par-relative delta on the win modal; wins submit {name, time, level, uid, par, features} to timed/{pushId} (features embedded per row - every timed board is unique, no meta bucket to join). Quick play has its OWN equation, PAR_MODEL_TIMED (between TIMED_PAR_MODEL markers in difficulty.js, refit-owned): timed rows exist only on WINS (losses never report), so the sample is win-censored and is never pooled with daily completions. The refit fits it as par-for-a-winning-run on handicap-adjusted times with priors centered on the daily posterior and a two-tailed outlier screen ([0.3x, 3x] predicted - the slow tail catches AFK rows); below TIMED_FIT_THRESHOLD=20 usable rows it ships as a verbatim copy of PAR_MODEL. predictPar selects the model on features.modeTimed (stamped by the timed path).
- **Daily:** One seeded puzzle per day, no levels. Modifiers are force-injected on improvement days when the target maps to a gimmick (sonar, wormhole, walls, mystery, liar, locked, mirror, compass) — see Adaptive Experimental Design. Falls back to the natural ~35% lottery only when the target is a non-gimmick feature (move-type counts, structural features).
- **Weekly:** ONE single canonical puzzle for the whole ET week (Monday → Sunday), one attempt per day for 7 days. Best time across the 7 attempts wins. Same bomb-hit mechanic as daily (info-value penalty: `+infoValue + BOMB_PENALTY_BASE` (3s), where info-value is the par-seconds of deduction the struck mine was anchoring, computed by `computeBombInfoValue` via a double solver run — see `src/logic/bombInfoValue.js`. NO re-fog: every other revealed cell stays revealed; the mine is preserved so adjacent numbers don't drop. The hit cell stays revealed with `cell.isStrike = true` so the strike sprite marks where the bomb was; strike cells count as flags for chording and persist across subsequent hits. Penalties live in the `bombHitEvents` log, not in wall-clock `elapsedTime`. Popup labels Minor/Key/Critical mines at info-value ≥ 2 / 6.5 / 13, quantile-anchored by `scripts/reanchor-bomb-tiers.mjs`. No game-over.) Board size 8–14 × 8–14, 2–4 modifiers stacked, daily-density. Architecture mirrors daily's canonical-board: `weeklyBoard/{weekStart}` is write-once on Firebase, pre-generated by `precompute-weekly-board.yml` Monday 00:00 UTC (`scripts/precompute-weekly-board.mjs`). Cloud-synced one-attempt-per-day cap at `users/{uid}/weeklyAttempts/{weekStart}/dayAttempts/{N}` so clearing localStorage can't grant a second try. Leaderboard is one row per player per week at `weekly/{weekStart}/{uid}` (set, not push) carrying `bestTime` and a `dayTimes` map. ## Privacy / sensor-permission posture

By design, GregSweeper requests ZERO sensor permissions and reads NO clipboard / camera / microphone / location data. Audited 2026-05-10:

- No `navigator.geolocation` calls anywhere.
- No `getUserMedia` (camera / microphone) calls.
- `navigator.clipboard.writeText` is used in `main.js` (Settings copy-link) and `diagnosticsModal.js` (copy diagnostic state). Write-only — modern browsers grant this without a permission prompt for user-initiated actions. No `clipboard.read*`.
- No `navigator.permissions.query()` calls.
- Notifications and persistent-storage are the only Web APIs that prompt the user, and both are gated by explicit opt-in actions (Settings toggle, install heuristic).

When adding a new feature, treat this list as a constraint: if it requires a sensor permission, surface that decision explicitly rather than silently expanding the permission surface.

## Greg-par Model (Daily)
Daily par is a linear regression over board features, fit in R offline against real completion data. Coefficients live in `PAR_MODEL` in `src/logic/difficulty.js`; the whole model lives in `src/logic/dailyFeatures.js` (`computeDailyFeatures`, `predictPar`, `breakdownPar`).
- **Move type, not move count, is the primary signal.** The solver in `boardSolver.js` classifies every deduction into one of five buckets at the three `totalClicks++` sites: `passAMoves` (trivial propagation), `canonicalSubsetMoves` (Pass B subset with larger-constraint size ≤3 — 1-1 / 1-2 / 1-1-1 shapes), `genericSubsetMoves` (Pass B, size ≥4), `advancedLogicMoves` (Pass C tank/gauss), `disjunctiveMoves` (Pass C with liar). Invariant: `passA + canonical + generic + advanced + disjunctive + 1 === totalClicks` for solvable boards. The solver still classifies all five, but the **model pools them for identifiability** (2026-06-08 rework): a VIF/sparsity audit showed the four raw move-types are too sparse / small-count to fit separately, so `predictPar` uses two earned tiers — **pattern** (`canonicalSubsetMoves + genericSubsetMoves`, subset deductions) and **search** (`advancedLogicMoves`, tank/gauss enumeration). `passAMoves` is dropped as a model predictor (it's a board-size proxy, VIF 7.4, absorbed into `cellCount`); `disjunctiveMoves` stays unmodeled (dropped 2026-05-04, confounded with `liarCellCount`). The solver keeps counting all five for diagnostics.
- **Stored feature vector (`dailyMeta`):** raw move-type counts + board shape (rows, cols, cellCount, totalMines, wallEdgeCount, etc.) + gimmick cell counts — UNCHANGED by the rework, so every historical board stays usable with no migration. The **model predictors are DERIVED from these** at predict/fit time: `cellCount` (lone size axis), `totalMines` (raw — VIF 3.6, fine once the redundant size features are dropped; density was tried 2026-06-08 but made the coefficient unreadable), `patternMoves`, `searchMoves`, `wallEdgeCount`, `zeroClusterCount`, and the modifier cell counts (kept split — sparse, prior-anchored until coverage missions build data). `nonZeroSafeCellCount` is dropped (near-redundant, VIF 23.7). One source of truth: `COEF_TERMS` in `dailyFeatures.js`, where each term is a function of the raw features (so the R refit derives the same from stored `dailyMeta`).
- **End-of-game modal** renders a per-term breakdown below par: `+18s search moves · +12s pattern moves · …`. Baseline terms (intercept + size + density) merge into a single chip to keep the line readable.
- **R refit workflow:** export Firebase JSON from the console → run `scripts/fit-par-model.qmd` → paste the emitted `PAR_MODEL = { ... }` block into `difficulty.js` → commit + deploy. The .qmd also runs diagnostics (residual plots, QQ for log-linear check, VIF for collinearity, bootstrap CIs).
- **Pre-ship scale anchor:** `scripts/calibrate-today.mjs` regenerates today's daily board locally and verifies `predictPar` lands inside the observed completion-time range before shipping new coefficients. One-board check, not a fit.

## Handicaps (user-specific par offsets)
- Each user has a handicap in seconds — their typical over/under vs Greg's CLEAN par across recent dailies, INCLUDING their bomb factor. Greg-par is clean board difficulty; the handicap holds everything about how the player actually plays, bombs included. Golf-style on the clean-skill part (negative = faster than typical); the bomb factor only ever adds time.
- Stored in `src/logic/handicaps.json`, a static JSON asset committed to the repo. Keyed by Firebase anonymous uid. Not in Firebase — shipping via static file avoids write-permission issues and lets handicaps update the same way everything else does (commit → GitHub Pages redeploy).
- Computed server-side (well, GitHub-Actions-side) by `scripts/refit-par-model.R` running daily: fits `brm(time ~ features + (1|uid))` with informative lognormal priors on each fixed-effect coefficient (centered on seed values, log-scale sigma = 1.0) plus `student_t(3,0,5)` on the between-user SD. Posterior means go to `PAR_MODEL`; the random-intercept posterior means are the clean-skill handicap after play-weighted recentering, then each player's mean per-play bomb cost is ADDED on top (so handicaps no longer sum to zero — the clean-skill part does, and the bomb factor lifts each one). Falls back to the mean-residuals path (clean-skill recentered + bomb factor added) when N or n_players is below the fit threshold.
- Client reads via `src/logic/handicaps.js`: `loadHandicaps()` fetches once, `getHandicap(uid)` returns the number or 0, `getHandicapsMeta()` returns `{updatedAt, modelFitN, nPlayers, method}`. Used in the end-of-game modal to show "Your par" alongside "Greg's Time" and in the diagnostics modal to surface what the user's device has cached.

## Refit Workflow (.github/workflows/refit-par-model.yml)
- Runs daily at 10am America/New_York (14:00 UTC; 9am ET in winter since Actions cron doesn't observe DST).
- Pulls `daily/*` and `dailyMeta/*` from Firebase (both world-readable — no secrets required), fits the Bayesian mixed-effects model via `brms` + Stan, patches `src/logic/difficulty.js` between `PAR_MODEL:START` / `PAR_MODEL:END` markers, and rewrites `src/logic/handicaps.json`. Commits both files under "Christopher Wells" identity and pushes to main, then explicitly dispatches the Pages deploy — GITHUB_TOKEN pushes never fire push-triggered workflows (GitHub's recursion guard), so without the dispatch each refit sat on main undeployed until the next human push (found and fixed 2026-06-10).
- Guards: `MIN_SCORES_TO_FIT = 30` (total scores required, priors do the regularisation so the old `MAX_COEF_DRIFT` clamp was retired), `MIN_PLAYS_FOR_FIT_INCLUSION = 5` (per-user TOTAL-play threshold — only users at/above this contribute to the global PAR_MODEL and receive a handicap entry in handicaps.json; below-threshold users get a provisional handicap computed client-side via `estimateHandicapFromHistory`. Set low because partial pooling shrinks a small-N user's random intercept toward zero and the heavy players anchor the fixed effects, so a light player can't drag the coefficients — the threshold only excludes true one-offs. Bomb-hit plays count toward the threshold now that they're included in the fit via the `bombHits` regressor), `ADAPT_DELTA = 0.99` with `MAX_DIVERGENT_FRAC = 0.25%` — a fit that comes back with Rhat > 1.05, ESS < 400, or more than 0.25% divergent transitions is rejected and the previous `PAR_MODEL` stays.
- Priors live in `PRIOR_MEANS` + `PRIOR_SIGMAS` at the top of the R script. Each per-coefficient prior is `lognormal(log(mean), sigma)`, naturally positive, with sigma = 1.0 giving roughly `[seed/2.7, seed*2.7]` at ±2 SD — wide enough for data to move a well-supported coefficient, tight enough to stop the collinearity-driven 10x swings the old lme4 fit produced at N=62.
- After the fit: random-intercept means are play-weighted-recentered (brms's sampler can park any overall baseline in either the global intercept or the random intercepts; without this recentering both users' handicaps come out shifted by the same ~100s constant). The recentered intercept is the clean-skill handicap; each player's mean per-play bomb cost (`BOMB_PENALTY_BASE`/hit for new-mechanic plays, the fitted legacy rate/hit for legacy) is then ADDED to it, so the handicap = clean offset + bomb factor and the set no longer sums to zero. The global intercept is separately bias-corrected so mean(clean-time-equivalent predicted par) = mean(actual time minus the bomb contribution) across the FIT POPULATION — keeping `predictPar` itself "clean-play par" (bombs live in the handicap, never in par).
- First CI run after a `brms` package cache miss takes ~5 min (Stan + dependencies compile). Subsequent runs hit the cache and finish in ~2–3 min including Stan model compilation.
- **Bomb-hit plays are kept in the fit; their cost is split — out of par, into the handicap.** Par stays clean: new-mechanic plays subtract their info-value penalty into `clean_time`, legacy +10s/re-fog plays carry their cost via the `legacy_bombs` regressor (fitted rate ≈ +14s/hit, surfaced in `handicaps.json` as `secPerBombHit`), so `predictPar` is clean-play par. The per-player bomb cost that par excludes is then folded back into THAT player's handicap (see Handicaps) — so bombs are a personal handicap component, never board difficulty. Kept-not-filtered replaced an earlier filter-out that dropped ~60% of plays with asymmetric bias (Chris's bomb-hit rate is higher than Kate's). Clients submit per-hit `bombHitEvents: [{ t, row, col, penalty, infoValue }, ...]`.
- New features added to the fit follow a threshold guard: until ≥20 plays have nonzero values for a given new feature, its coefficient is forced to 0 in the SHIPPED `PAR_MODEL` even if brms has fit a posterior. This avoids the lognormal-prior median (~prior_mean × 1.65) being shipped as a real coefficient, which would inflate `predictPar` with no data justification. The fit still treats it as a regressor so other coefficients aren't polluted by missing-data variance.
- `scripts/refit-par-model.R` is the full pipeline; `scripts/fit-par-model.qmd` is the exploratory Quarto notebook (diagnostic plots, VIF, bootstrap CIs) for interactive review.

## Canonical Daily Board (cross-client board agreement)
Every player on the same ET date plays the same board, period. Three divergence sources are eliminated:

1. **Code-version drift** between clients on different cached SW bundles. Same seed + different code = different board. Closed by storing the resolved board on Firebase, not the seed.
2. **Experiment-target drift** between clients with different cached `experimentTarget.json`. Same code + different target = different `selectDailyRngSeed` winner = different board. Same fix.
3. **Firebase cold-load race** where `loadDailyBoard` returns null because the Firebase SDK hasn't initialized yet, and the client silently falls through to local generation against a stale `experimentTarget.json` — different `:trialN` winner from whatever wrote the canonical, so same date but divergent board. Closed by `runStartupGate` in `main.js`, which renders nothing user-interactive until the SW is current, Firebase is ready, and today's canonical is in memory. Without this gate, a player who lost the cold-load race would silently play a divergent board (this happened to Kate on 2026-05-06 v1.5.30 — the trigger for the gate).

Mechanism:
- **`dailyBoard/{date}`** in Firebase Realtime DB stores the fully-resolved board (mine layout, gimmick assignments, displayed numbers, wall edges) — write-once at the rules layer.
- **`getLocalDateString()`** is anchored to America/New_York via `Intl.DateTimeFormat('en-CA', { timeZone: 'America/New_York' })`. A player in Tokyo loading at 9am JST sees the puzzle for the previous ET date if it's still before midnight ET.
- **`src/firebase/dailyBoardSync.js`** — `loadDailyBoard(date)` / `saveDailyBoard(date, payload)` / `serializeBoard({ board, rows, cols, ... })` / `deserializeBoard(raw)`. Both load and save now go through `waitForFirebaseReady` (50ms-poll up to 8s) so a cold-load race can't make them silently no-op. The serialiser drops `false`-valued booleans to keep the JSON tight (~30% smaller), and the deserialiser fills them back in from defaults so the consuming code sees a fully-shaped cell object.
- **Startup gate (`runStartupGate` in `main.js`)** — runs before `init`'s routing branch, behind the `#boot-overlay` div in `index.html`. Three steps: (1) `ensureLatestServiceWorker` — `registration.update()` and wait up to 3s for any new SW to activate (controllerchange handler in `<head>` reloads us if it does), (2) wait up to 8s for `firebase.apps.length > 0` and stash the result in `state.firebaseReady`, (3) pre-fetch `loadDailyBoard(today)` into `state.canonicalDailyBoard = { date, raw }`. Practice loads (`?seed=`) skip step 3. Boot overlay only hides at the END of `init` (after routing + first newGame await), so the player never sees a flash of a wrong board.
- **Daily entry path** (`gameActions.js` daily branch — `async`): read `state.canonicalDailyBoard` first; if it exists and matches `state.dailySeed`, deserialize verbatim (skip generation, gimmicks, solvability retry). Fall back to a fresh `loadDailyBoard` call only when the cached canonical is for a different date (tab open across midnight) or when the gate's fetch failed. If both miss, run the full local-generation flow and `saveDailyBoard` the result fire-and-forget so the next visitor sees what we generated. The solver still runs locally on the resolved board to compute features (move-type counts) and the best-start cell — that's deterministic, so different clients agree on features even if they came in via different paths.
- **Completion ↔ cloud reconciliation** in `runStartupGate` (one `daily/{today}` read, both directions; row matching via the pure `src/logic/scoreRowMatch.js`, node-tested): (a) local "completed" flag set + the account's cloud row POSITIVELY divergent (truthy `rngSeed` ≠ canonical) → clear `minesweeper_daily_completed_date` plus the cached par/moves keys, unblocking the canonical replay (a missing row trusts the local flag — clearing on missing-score used to let raced uid lookups unlock replays); (b) local flag unset + a cloud row matching the canonical's effective seed → ADOPT the completion (markDailyCompleted + drop any in-progress daily save), so finishing the daily on one device marks it completed on every device sharing that account. The `?mode=daily` deep link gates on `isDailyCompleted` like the Daily card (a notification tap can't relaunch a completed daily). Skipped on the test env and for `?seed=` practice.
- **Submission dedupe** in `_doSubmitOnlineScore`: before pushing, read the date's rows and skip the push (returning `'duplicate'` — a third outcome alongside `true`/`false`) when this uid already has a row for the SAME effective board seed. Matching is per BOARD, not per uid: rows omit `rngSeed` when it equals the dateString, so the effective seed is reconstructed (`effectiveRowSeed`), practice rows can never block the real daily, and a divergent historical row doesn't block a canonical replay. First completion wins: callers toast "Already on the board from another device" and skip the `dailyHistory` write on `'duplicate'`; the offline retry queue treats `'duplicate'` as resolved (covers the offline-race where another device submits before the queue flushes). Both submit paths (auto in winLossHandler, manual in main.js) anchor the score's date to `state.dailySeed` (the board's date), so a finish at 12:00:01 AM lands on the board's own leaderboard.
- **Resume eligibility** (`src/logic/resumeEligibility.js`, pure + node-tested; consumed by `tryResumeGame` in `src/game/gamePersistence.js`): a persisted save resumes only if (a) its `gameMode` matches the slot it was loaded from, (b) a daily save carries BOTH `dailySeed` and `dailyRngSeed` and the seed equals today's ET date (practice `?seed=` equals its custom seed instead), (c) the save's `dailyRngSeed` matches the canonical board's when one is cached for that date (divergent-canonical drop), (d) a weekly save carries full identity (`weeklySeed`/`weeklyDay`/`weeklyRngSeed`) matching the live ET clock, and (e) cells have `row`/`col` (v1.5.19 corruption check). Date anchors compare against the CLOCK (`getLocalDateString`/`getWeekStart`/`getWeekDayIndex`), never against live state fields — a session that survives midnight ET carries yesterday's seeds in state. Saves missing their seed identity are REJECTED, not waved through: the pre-fix Daily card handler nulled live seeds before `switchMode` persisted the outgoing game, and those fingerprint-less snapshots resumed yesterday's board as "today's" daily (found 2026-06-12).
- **Date-rollover discipline (2026-06-12):** daily/weekly identity is derived inside `newGame` from the ET clock (practice keeps its caller-set seed via `isDailyPractice`); the mode-card handlers no longer pre-set identity fields before `switchMode` (mutating identity before the outgoing game is persisted forges or strips the snapshot's fingerprint — the root cause above). A live session that sleeps through midnight (background tab, suspended PWA) is caught by `expireRolledOverGame` in `main.js` on `visibilitychange → visible`: `isLiveGameExpired` (same pure module) checks the live game's date anchor against the clock; on expiry the persisted save is cleared, `state.status = 'expired'` (persist only writes playing/idle, so the dead game can't re-save), and the player lands on the title screen with a toast. Actively-visible play across midnight is deliberately NOT interrupted — finishing keys under the board's own date.
- **codeVersion provenance**: SW posts `{ type: 'codeVersion', value: CACHE_NAME }` to clients on activate, plus replies to `getCodeVersion` requests. Page caches the result in `state.codeVersion` and stamps it onto canonical-board writes (replaces the stale `'v1.5.19'` literal that used to lie about which build wrote a board).
- **On-demand par calc** in `main.js` (leaderboard modal opening before play) calls `loadDailyBoard` directly — also benefits from `waitForFirebaseReady` now.
- **Practice daily** (`?seed=...`) bypasses canonical fetch/write — practice is per-user and shouldn't pollute the shared bucket.
- **Pre-generation** runs nightly at 00:00 UTC via `.github/workflows/precompute-daily-board.yml` (`scripts/precompute-daily-board.mjs`). Anchoring to UTC gives a 4-5h buffer before the daily flips at 00:00 ET. The seed-to-board pipeline lives in `scripts/daily-board-pipeline.mjs` (shared with the regeneration tool, ONE copy) and runs the SAME pipeline `selectDailyRngSeed` + `gameActions.js` would: load the experiment target, force-inject the gimmick if applicable, run candidate selection across 10 trial seeds, pick the winner, write board + dailyMeta to Firebase. Idempotent — write-once rules silently no-op repeated writes for the same date.
- **Board cache is network-first with cache fallback** (`loadDailyBoard`/`loadWeeklyBoard` + `src/firebase/boardCache.js`): while online, every load fetches the canonical and refreshes the local copy (~2KB); offline, the cached copy keeps daily/weekly playable. The cache is deliberately NOT authoritative — an admin regeneration can replace an unplayed future board, and a blindly cache-first client that had prefetched the old layout would play a divergent board on the day. Server-reachable-and-empty returns null (no resurrection of deleted canonicals); admin deletions must always be followed immediately by a rewrite.
- **Force-regeneration of an UNPLAYED future date**: `scripts/regenerate-daily-board.mjs` via `.github/workflows/regenerate-daily-board.yml` (`-f date=YYYY-MM-DD -f dryRun=true|false`). Service-account auth bypasses write-once; replaces dailyBoard + dailyMeta with a fresh run of the shared pipeline; prints gated AND ungated certification verdicts. Rails: refuses non-future ET dates (`--force-past` to override) and any date with score rows. Built for the 2026-06-14 reveal-gating re-certification. Weekly equivalent: `regenerate-weekly-board.yml` (also wipes leaderboard + attempt markers).
- **One-off bootstraps** (e.g., a date played by a real user before the canonical-board ship): `scripts/bootstrap-daily-board.mjs` — hardcode date + seed + force-injection state, regenerate, write. Used to lock 2026-04-27 to Chris's actual played v1.5.18 board.
- **Audit script**: `scripts/audit-divergent-scores.mjs` reads `daily/*` and `dailyBoard/*` and reports any score whose `rngSeed` mismatches the canonical's for that date. Read-only by default; `--delete` flag does the cleanup. Used to confirm no historical divergent rows remain after the gate ships.

## Adaptive Experimental Design
- EVERY daily generates 10 candidate seeds (`${dateString}:trial0`..`trial9`), solves each, scores each by its assigned mission, and picks the highest-scoring candidate. ~500–800 ms of CPU on first daily load.
- **Multi-objective per-slot missions.** Slot 0 = the PRIMARY mission, force-injecting the high-CV target's gimmick (chosen by the R refit) and allowed to roll a second gimmick at the natural ~10% rate. Slots 1–9 = COVERAGE missions, each force-injecting a different undersampled gimmick from the ranked `coverage_targets` list (also produced by the refit) and constrained to single-gimmick only via `getDailyGimmick(seed, rng, forcedGimmick, singleOnly=true)`. Slots cycle through the coverage list if it's shorter than 9 entries.
- **Scoring.** Each candidate's score is `target_count_in_features × deficit_weight`, where the target and weight come from `getMissionForSlot(i)` in `experimentDesign.js`. Slot 0's weight is fixed low (`PRIMARY_WEIGHT = 0.1`) so it only wins when its target's cell count is high enough to overcome the gap with the heaviest coverage weight (typically ~0.5 for the most undersampled gimmick). In practice this yields roughly 1-in-10 primary outcomes when the coverage list is well-populated — matching the design intent that 9 of every 10 dailies probe sample-size deficits, not posterior-uncertainty.
- **Force-injection rationale.** Without forcing, the natural per-seed gimmick-inclusion rate of ~6.6% means ~50% of seeds have zero of the targeted feature, making maximisation meaningless. Force-injection drops miss rate to 0% so the 10-way contest competes on cell COUNT and DEFICIT WEIGHT instead of mere presence. All four call sites of `getDailyGimmick` (`selectDailyRngSeed`, `gameActions.js`'s play path, the on-demand par calc in `main.js`, and the precompute script) pass the same `(forcedGimmick, singleOnly)` resolved from `getMissionForSeed(seed)` so the chosen seed and the regenerated board agree.
- **Primary-target selection.** Same as before: after each fit, the coefficient with the highest posterior coefficient of variation (SD / |mean|) from a whitelist of push-able features wins — EXCLUDING any feature targeted in the last 3 days (`recentTargets` in `experimentTarget.json`). Stops three liar-heavy dailies in a row while still pushing data toward the most uncertain coefficient. Client fetches via `loadExperimentTarget()` in `src/logic/experimentDesign.js`.
- **Coverage-target selection.** The R refit counts unique-date occurrences of each gimmick feature, drops any gimmick at >= 20 boards (COVERAGE_SATURATION_BOARDS coverage throttle - saturated features still appear via the primary target or the natural lottery) in `df_fit`, computes `deficit_weight = 1 / (count + 1)` per gimmick, excludes the chosen primary target, and emits the result sorted descending by deficit. So `liarCellCount` with 1 board (deficit 0.5) ranks above `compassCellCount` with 5 boards (deficit 0.17). Lives in `experimentTarget.json` as `coverage_targets: [{feature, n_boards, deficit_weight}, ...]`.
- **Double-gimmick rate.** Default `DOUBLE_GIMMICK_PROB = 0.10` (was 0.20). The coverage missions absorb most of the "make boards more varied" goal so we don't need the natural double rate doing as much work, and lower noise = cleaner per-feature deltas. Coverage slots additionally suppress the roll entirely via `singleOnly`.
- **State plumbing.** `state.dailyRngSeed` holds the effective RNG seed (the trial variant on improvement days, plain dateString otherwise). All daily-mode RNG creation in `gameActions.js` routes through this field; `state.dailySeed` remains the plain date for Firebase keys / leaderboard joins / local-storage lookups. The play path uses `getMissionForSeed(state.dailyRngSeed)` to recover which mission won, then force-injects the matching gimmick — without this, a coverage-slot winner would get the primary gimmick force-injected on the play board and we'd ship the wrong layout.
- **Determinism.** Same date + same loaded primary + same coverage_targets list → same per-slot missions → same candidates → same winner across all clients. Cache-version drift on `experimentTarget.json` between clients can desync the chosen seed; Check-for-Updates handles most cases, and the diagnostics modal surfaces the loaded primary target + coverage list so mismatches are inspectable.

## Challenge Difficulty Curve (Sawtooth)
Computed by `getDifficultyForLevel()` in `difficulty.js` — no static table.
- **L1-10:** Tutorial ramp, 5x5→9x9, 8%→16% density, no gimmicks
- **L11-90:** Each 10-level block introduces one gimmick. Board drops to 11x11 at intro, ramps to 14x14 by block end. Density drops 10% (relative) at intro, ramps to next peak.
- **L91-120:** Final 30-level ramp from 11x11 to 14x14, density reaches 34%.
- **Gimmick selection:** Primary gimmick is 100% present during its intro block. Old gimmicks appear as secondary (60%) and tertiary (10%). By L120: guaranteed 3 gimmicks.
- **Hard cap:** 34% mine density maximum for fast board generation.

### Theme Expression Tokens (the "wildly different per theme" layer)
The `:root` block in `global.css` exposes a large token contract so a theme can diverge in **motion, type, texture, cell treatment, and feedback colors**, not just palette. Every token defaults byte-for-byte to the value that used to be hardcoded, so a theme that overrides nothing renders exactly as before. Groups:
- **Motion:** `--anim-reveal-dur/-ease`, `--anim-flag-dur/-ease`, `--anim-mine-dur/-ease`, `--ease-bounce`, `--ease-standard`. Only the `animation` shorthand (duration/easing) is tokenized — `@keyframes` interiors can't read vars, so a theme changes *choreography* by shipping its own keyframes in its theme file. History note: the flag/mine tokens were declared but DEAD (`.cell.flag-pop` / `.cell.mine-chain` hardcoded their literals) until 2026-06-10; the root defaults were corrected to the effective literals (flag 0.35s, mine 0.4s) when wired, so tokenless themes render byte-identically while the ~17 themes that had set flag tokens finally animate as designed. Flag choreography overrides use `[data-theme=X] .cell.flag-pop { animation-name: ... }`.
- **Type scale:** `--fs-title`, `--fs-lcd`, `--fs-cell-number`, `--fs-mine`, `--fs-flag`.
- **Cell + sizing:** `--cell-shape-radius`; `--cell-min-size` / `--cell-max-size` / `--cell-fit-max-size` are read by JS in `boardRenderer.js` (`resizeCells`/`adjustCellSize`) because JS owns `--cell-size` at runtime — a CSS `--cell-size` override would be clobbered, so these bound tokens are the only way a theme can widen/tighten cells. `--cell-gap-seal` = the grout color painted between cleared cells when an effect is on (see Effect layering). It defaults to `--color-cell-revealed`, but **that default is the banned "slab" look (grout == tile) and must NEVER ship on an effect-bearing theme** — see the grout-contrast HARD rule below. Every theme that registers a `THEME_EFFECTS` entry sets its own contrasting grout (forest sage, candy pink, circuit-board copper trace, comic black ink-gutter, inferno lava crack, stained-glass dark lead, etc.).
- **Backgrounds:** `--bg-page-image` (a fixed `body::before` layer, default `none`), `--bg-page-blend`, `--bg-board` (`#board-container`). Vector only (gradients / CSS patterns / SVG data-URIs) — no raster assets, so no `sw.js` ASSETS/CACHE_NAME coupling. Animated / board ambiance routes through `src/ui/themeEffects.js`, not CSS. **Effect layering:** the `.theme-fx` container is `z-index: 1` — between unrevealed cells (z-index auto/0) and revealed cells (`.cell.revealed` is `z-index: 2`) — so ambient animations (rain, leaves, sparks, stars) play over the fog + background gaps but never wash over revealed numbers/markers. Walls stay on top at `z-index: 3`. Each theme's effect should keep particle counts modest but the particles large/visible (the original Forest spores were 2-4 sub-5px fading dots = invisible; falling leaves at 13-22px read clearly). **Gap-seal occlusion:** lifting revealed cells above the effect handles the cell *bodies*, but the 1-2px inter-cell *gaps* would still show the effect as slivers. `applyThemeEffects` adds `.fx-on` to `#board` only when a particle effect is registered (effect-less themes are byte-identical — no seal, no z-index lift). As of wave 2 (2026-06-10), 24 of 26 themes register an effect; only classic and dark stay deliberately bare (restraint is their identity, like classic's default synth voice). Wave 2 effects (editorial ink flecks + headline rules, sumi-e ink motes + brush wisps, blueprint drafting ticks + measure lines, cartography route dashes + sounding rings, origami gliding paper triangles, comic halftone clusters + speed lines) are VECTOR-DRAWN — styled divs with gradients / clip-paths / dash patterns, never emoji glyph particles — per the "worlds apart, less emoji" direction; prefer that style for any future effect work. Under `.fx-on`, each revealed cell gets a two-layer seal: a `::before` SQUARE solid fill of `--cell-gap-seal` over cell+gap (z-index -2; square so four neighbors tile every 4-cell junction shut — a rounded ring or box-shadow leaves a diamond there, the inherent weak spot of rounded corners), and a `::after` rounded `--color-cell-revealed` body on top (z-index -1) so each tile shows its own surface and the grout reads only in the gaps. The seal EXCLUDES the face-cue markers — sonar/compass/wormhole (translucent bg tints), locked/pressure-plate/defused (`::after` glyphs), and MIRROR (dashed border): the fx-on z-lift makes each revealed cell a stacking context, and negative-z pseudos paint ABOVE the element's own background and borders, so a sealed mirror cell's dashes were buried (found 2026-06-11, the mirror daily on chalkboard — invisible pairs in prod). The tint markers get an opaque revealed base composited under their tint on `.fx-on` so the effect can't bleed through them; a mirror PAIR is adjacent, so the seam between the two excluded cells can show a 1px effect sliver (accepted). Liar (outline — outlines paint last) and mystery (text) survive the body layer and keep the full seal. Colorblind mode (whose `::after` is a shape glyph) forces `--cell-gap-seal` back to the revealed color so the tile reads as one cleared surface. Verified by dyeing the effect a unique color and pixel-sampling every gap + junction: 0 leak. **Holistic fog art (2026-06-11):** `#board` consumes `--bg-fog-art` (default `none`) as a background-image sized 100% x 100% under the cells; a theme sets it to ONE large vector drawing (data-URI SVG or gradient) and switches its `.cell.unrevealed` gradient + `--color-cell-hidden-hover` to TRANSLUCENT rgba tints, so the drawing reads through the fog as one continuous image spanning the board (revealed cells stay opaque -- revealing tears the sheet). Compositing base is `#board`'s background-color (`--color-border`); the gallery mini-boards mirror that. Seven worlds use it: origami (crane crease pattern), cartography (worn chart), comic (inked page layout -- part of the 2026-06-11 comic rebuild), blueprint (cyanotype sheet + white number set), chalkboard (half-erased workings + chalk-dust number text-shadow), noir (venetian-blind bands), ocean (sounding chart). Fog must STILL read as interactive tiles: keep borders/hover and tint alphas around 0.3-0.55, and drawings need bolder strokes than instinct suggests (the fog tint sits on top).
- **Special-cell markers:** `--marker-liar-outline(-width)`, `--marker-sonar-tint`, `--marker-compass-tint`, `--marker-wormhole-0/1/2`, `--marker-mirror-0/1/2(-width)`, `--marker-plate-timer/-border/-glow`. See legibility contract below.
- **Overlays / power-ups / timer / victory:** `--overlay-*` (wrong/correct flag, suggest move/start, mine-hit glow), `--pu-*` (shield, xray, lifeline, magnet), `--timer-warning/-critical/-critical-glow`, `--victory-color/-duration`.

**Special-cell legibility contract (HARD FLOOR — do not relax):** marker tokens may be re-hued slightly by a theme only to avoid clashing with its own background. Every marker must stay (a) unmistakable as its modifier, (b) mutually distinct, (c) high-contrast vs the cell bg (line markers ≥ 3:1, like `--color-wall`), (d) above the alpha floors (tints ≥ 0.12, borders ≥ 0.6). Identity is anchored in a non-color channel (📡/arrow glyph, italic+underline, dashed border vs outline vs fill) so meaning never rests on hue alone. The markers keep their `!important` so an aggressive cell-bg can't bury them.

**Grout-contrast contract (HARD RULE — absolute, do not relax):** for any theme that registers a `THEME_EFFECTS` entry (so `.fx-on` is set and the gap-seal is painted), `--cell-gap-seal` MUST be a color that visibly CONTRASTS `--color-cell-revealed` — never equal to it. Grout == tile produces a solid "slab" where cleared cells merge into one undifferentiated block; this was explicitly rejected ("I see that we might be making grout fill that's the same as the tiles. Absolutely don't do that."). Cleared cells must always read as separate tiles set in a contrasting grout (chalk line on slate, copper trace on PCB, black ink-gutter between comic panels, dark lead between stained-glass panes, glowing lava crack between charred inferno tiles). Practical floor: luminance delta ≥ ~50 between grout and tile, and the grout should be thematically motivated (it's the "what's between the tiles" of the world), not an arbitrary outline. The default (`var(--color-cell-revealed)`) is intentionally the slab so an *effect-less* theme stays byte-identical — but it is a bug on any effect theme. Verify by revealing a region and confirming the gap lines are clearly a different color from the tile bodies. Colorblind mode is the one exception: it forces gap-seal back to the revealed color (the `::after` shape glyph carries identity instead).

**Per-theme objects + moments contract:** every theme defines its game objects and key moments in its own idiom, not just palette. (a) **Objects** live in `THEME_UNLOCKS` (`themeManager.js`) as emoji fields: `mine`, `flag`, `strikeCell` (the detonated cell — falls back to `mine` then `💣` if unset), plus `smiley`/`smileyWin`/`smileyLoss`. `getThemeEmoji(type)` in `boardRenderer.js` resolves them (emoji-pack override > theme). Architected so hand-drawn SVGs swap in later via the same classes with zero JS rewiring. That swap is now live as **`THEME_SPRITES` in `src/ui/spriteLoader.js`** (theme -> {mine, flag, strikeCell} -> `assets/sprites/themes/<theme>-<object>.svg`): `getSpriteUrl` consults it before the Tier 1 canonical-emoji match, and a theme sprite renders ONLY when the resolved emoji equals that theme's own object emoji -- so a player emoji-pack override always wins and undrawn themes keep their emoji verbatim. Batch 1 (2026-06-11) ships editorial / sumie / blueprint / cartography / chalkboard / splitflap / galaxy / circuitboard / comic full sets plus noir's strikeCell (the crime-scene chalk body outline). All 28 files are in the sw.js ASSETS precache. Ship future worlds by adding files + one registry line. (b) **Reveal moment:** each theme overrides `.cell.num-pop, .cell.revealing { animation-name: <theme>Reveal }` with its own `@keyframes` in its theme file (editorialPrint, sumieBleed, candyPop, chalkScrawl, noirFlicker, glassBloom, flapFlip, circuitPower, comicPop, …). (c) **Win moment:** per-theme confetti palette in `effectsRenderer.js` (the `themeColors` map inside `showConfettiBurst`, keyed by `state.theme` — beware: a palette under a stale key silently never fires; sakura's sat under its pre-rename key 'cherry-blossom' until 2026-06). (d) **Explosion moment:** `showExplosionBurst(originX, originY, count)` in `effectsRenderer.js` fires a radial, theme-colored detonation at the hit cell from `chainRevealMines` (challenge/timed loss). It reuses `showConfettiBurst`'s particle engine via physics `opts` (the confetti defaults are unchanged). A new theme must supply, at minimum, its objects + a contrasting grout (if it has an effect) + a confetti palette entry.

**Regression invariant + how to verify:** because every default equals the old literal, all themes are byte-identical until one deliberately overrides a token. The frozen computed-style probe is a COMMITTED TOOL: `scripts/theme-style-probe.html` (serve the repo with `python -m http.server 8080`, open the page, it hashes synthetic marker/overlay/timer/cell elements across all 26 themes; capture `window.__probeResults` on the BASELINE checkout, then paste into the changed checkout's page and Diff). Implementation notes baked into the tool: animations are frozen via `animation-play-state: paused` at delay 0 — NOT `animation: none`, which would erase `animation-name`/`duration` from computed styles and blind the probe to choreography identity; stylesheet links are cache-busted per run (a same-browser baseline would otherwise diff cached CSS against itself and report "no differences" for real changes); `.num-pop`/`.revealing` probe elements carry reveal-choreography identity; keyframe INTERIORS remain by-eye review. Workflow for a theme wave: `git stash push` the theme edits → baseline → `git stash pop` → diff; only deliberately redesigned themes may appear.

**Grandfathered carve-outs (don't "fix" blindly):** ~15 themes use `background-clip: text` gradient titles and `.modal` uses `backdrop-filter: blur(4px)` — both pre-existing and intentional; keep them but don't add *new* gradient-clip titles or decorative glass to showcase themes that lack them. Orbitron + Fredoka webfonts are loaded in index.html and used by aurora/galaxy/candy as real `--font-display` values (not dead). Two parallel timer color systems exist (`.lcd-display.timer-*` and `.timer-*` with `!important`); both now read `--timer-warning`/`--timer-critical`, whose defaults are the effective `!important` colors (`#ff9800` / `#f44336`).

  - `dailyMeta/{dateString}` — per-date board features (write-once, public-read) — `{ features: {...}, writtenAt }`. Read by the R refit.   - `dailyBoard/{dateString}` — canonical board layout for the date (write-once, public-read) — `{ rows, cols, totalMines, rngSeed, codeVersion, activeGimmicks, gatedCert?, cells: [{isMine, adjacentMines, displayedMines, isLocked, isMystery, ...}, ...], wallEdges?, writtenAt }`. `gatedCert: true` marks boards certified under the reveal-gated solver contract (see No-Guess Contract); its absence means the historical ungated contract. The source of truth for "what board does today look like" — `gameActions.js`'s daily branch fetches this BEFORE local generation so every player on the same ET date plays the exact same layout regardless of cached code or experimentTarget. Written by either the GitHub Actions pre-gen workflow at 00:00 UTC or, as fallback, by the first client that loads on a fresh date. Each cell ships with both `adjacentMines` AND `displayedMines` so future changes to wall-aware adjacency or gimmick-display logic can't retroactively shift historical numbers. ~1-3 KB per day.
  - `users/{uid}/` — cloud progress sync (maxCheckpoint, dailyStreak, bestDailyStreak, lastDailyDate)
  - `users/{uid}/dailyHistory/{dateString}` — per-user completion record (`time, par, delta, submittedAt`). Written on every daily completion, read on every leaderboard-modal open for the SVG history chart.
  - `users/{uid}/friends/{friendUid}` — mutual friend list entry (`name, addedAt`). Owner-only READ (lists are private). The write rule carries a deliberate exception: a stranger may write/delete ONLY the entry keyed by their own uid (`auth.uid === $friendUid`) — that is what makes one code redemption link BOTH sides. Self-friendship rejected in validate. Accepted tradeoff: anyone who learns a uid could self-insert into that list (visible in the Friends tab, removable, fine at this community's scale).
  - `friendCodes/{CODE}` — ephemeral friend codes (`uid, name, createdAt`), CODE = 6 chars of `A-Z2-9` (no 0/O/1/I/L). 15-minute life enforced SERVER-SIDE by the rules read gate (`createdAt > now - 900000`); multi-use within the window BY DESIGN (one code on a classroom projector serves the whole class); create-only-if-absent (no hijacking); anyone authed may delete (cleanup). The 3-hourly scrub workflow deletes codes older than 1h. test/friendCodes.test.mjs pins the TTL + regex against src/logic/friendCodes.js; test/friendsRules.test.mjs pins the rule structure.
- Anonymous auth (`firebaseProgress.js`): silent sign-in on load, no UI. `saveProgress` calls before auth completes are coalesced into a pending-save and flushed once `_ready` flips, so fast daily completions on slow connections don't drop their cloud sync. **The uid is reactive — see Account Linking below.** Modules that depend on per-user data (push subscription, daily-residuals cache, settings UI render) subscribe via `subscribeToUidChanges` so they reload when the user signs in on a second device.
- Cloud sync: saves on checkpoint advance + daily completion, loads on init. Checkpoint takes the max. Daily streak is date-anchored: cloud date > local date adopts cloud's streak AND date verbatim (even if streak went down — the most recent play has the latest info), same date takes the higher streak, cloud stale keeps local. `bestDailyStreak` is always the high-water mark.
- Falls back to localStorage leaderboards if Firebase unavailable
- Security rules live in `firebase-rules.json` and are AUTO-DEPLOYED on every push to main via `.github/workflows/deploy-firebase-rules.yml`. (`firebase deploy --only database`, config in `.firebaserc` + `firebase.json`, is the manual equivalent.) The file IS the live ruleset; edits to it ship.
- **Timestamp fields validated `=== now` (e.g. `daily/*/timestamp`, `users/*/dailyHistory/*/submittedAt`) MUST be written with `firebase.database.ServerValue.TIMESTAMP`, never a client `Date.now()`.** The server sentinel is the only value that equals server `now`; a client number is silently rejected and the whole write dropped. This froze every stats page from 2026-05-25 to 2026-06-08, when commit 866683d swapped the sentinel for `Date.now()` on the dailyHistory write (to make the offline queue serializable) and the rule rejected every completion after.

## Account Linking (cross-device sign-in)
The default anonymous uid per device makes phone and PC look like two unrelated players. `src/firebase/firebaseAuth.js` adds Google and email-link sign-in so the user can upgrade the anonymous account on one device and sign in to the same identity on another — both devices then share the same uid and `users/{uid}/*` syncs automatically (streak, dailyHistory, weeklyAttempts, checkpoints, handicap).

- **Provider 1 (Google):** `linkWithGoogle()` calls `auth.currentUser.linkWithPopup(GoogleAuthProvider)`. On the first device this upgrades the anonymous account (uid unchanged). On a second device the credential is already attached to the first device's uid, so Firebase throws `auth/credential-already-in-use`; the module catches it, shows a confirmation modal (delegated to the `onCredentialConflict` callback supplied by `main.js`), then calls `signInWithCredential` to SWITCH the second device's session to the existing uid. The abandoned anonymous Auth record is `.delete()`d as part of the switch so it doesn't linger in Firebase Auth.
- **Provider 2 (Email link / passwordless):** `sendEmailLink(email)` → Firebase emails a one-time link → `tryCompleteEmailLink()` (called at boot in `init()` once auth has settled) finishes the sign-in or switch. The originating device stashes the email in `localStorage['minesweeper_pending_email_link']`; on a different device the boot flow calls the `promptForEmail` callback so the user re-enters their email (Firebase security requires the email to be confirmed on the completing device).
- **Apple sign-in is intentionally deferred.** It requires an Apple Developer Program enrollment ($99/yr), a Service ID in Apple Developer, and a private key uploaded to Firebase. Can be added later without disturbing the existing flow.
- **The uid is reactive at runtime.** `firebaseProgress.js`'s anonymous-auth init now subscribes to `onAuthStateChanged` rather than capturing the uid once. `subscribeToUidChanges(callback)` fires whenever the uid flips. Callbacks distinguish `isInitial: true` (first auth settle of the session, no in-memory state to clear) from `isInitial: false` (mid-session switch — old uid's queued writes are discarded, push token is re-minted under the new uid via `initPushAuthListener` → `refreshTokenIfStale`, and the daily-residuals cache is rebuilt from the new uid's Firebase history).
- **V1 scope is intentionally simple:** signing in on the second device ABANDONS that device's anonymous data — both leaderboard rows under the old uid and the `users/{old_uid}/*` subtree stay in Firebase but are no longer reachable from the player. The confirmation modal warns the user before the switch. No merge logic (no taking max across two uids' streaks/checkpoints, no rewriting old leaderboard rows). If we ever want merge, it's additive — add a pre-switch snapshot step in `firebaseAuth.js`'s conflict handler and write merged values into `users/{new_uid}/*` after the credential switch.
- **Push subscription handoff.** When the user signs in (uid switches), `firebaseAuth.js` first writes `users/{old_uid}/pushSubscription = null` while still authed as the old uid, so the hourly send-push cron stops firing to the abandoned device. `firebasePush.js`'s `subscribeToUidChanges` listener then calls `refreshTokenIfStale` on the new uid — if that uid has `notificationPrefs.enabled === true`, a fresh FCM token is minted and written under the new uid (overwriting any existing token), so pushes follow the player to the device that just signed in. If the new uid hasn't enabled notifications, the device stays unsubscribed until the user toggles it on.
- **Manual Firebase Console prerequisites (one-time, before this feature works):**
  - Authentication → Sign-in method → enable **Google** (no extra cost).
  - Authentication → Sign-in method → enable **Email link (passwordless sign-in)**. The action-URL is computed dynamically (`?emailLink=1` on the current origin) so test and prod both work without further config.
  - Authentication → **Settings tab** → Authorized domains → add `christopherwells.github.io`. Anonymous auth doesn't need this (server-side credential issuance) but Google sign-in's popup-return DOES, otherwise the popup throws `auth/unauthorized-domain`. Firebase auto-adds `localhost` and `<project>.firebaseapp.com`; the GitHub Pages domain has to be added manually.
  - No security-rule changes needed for V1 (the existing `users/$uid/*` and `daily/$date/$entry` rules continue to work — sign-in just changes which uid the device authenticates as).
- **Verifying via diagnostics:** the Diagnostics modal (`?debug=1` then Settings → Show Diagnostics) shows a "Signed in" row reporting provider + email. If two devices report the same uid and email, the link worked.

## Leaderboard Name Filtering (hate speech)
Two-layer filter so slurs don't reach the daily/weekly leaderboards. Scope is **hate speech only** (racial, ethnic, homophobic, transphobic, misogynistic slurs) — NOT general profanity, to keep false positives near zero.

- **Client reject** (`src/logic/nameFilter.js` + `hateSpeechTerms.js`): `setPlayerName` (statsStorage.js) calls `containsHateSpeech`, which normalizes (lowercase, leetspeak map `4→a 3→e 1→i 0→o @→a $→s …`, strip non-letters, collapse 3+ repeats to 2) then substring-matches the **client-safe** term subset (`HATE_SPEECH_TERMS_CLIENT_SAFE` — only terms that don't collide with innocent words, so a dumb `includes()` won't reject "Nigeria"/"Pakistani"/"RaccoonKing"/"SanDiego"/"Fagan"). `setPlayerName` returns `{ ok, reason }`; the Settings name input toasts "Please choose a different name" on the `change` event (not per-keystroke) and reverts to the last good name. Rejected names are never saved.
- **Server sweep** (`scripts/scrub-leaderboard-names.mjs` + `.github/workflows/scrub-leaderboard-names.yml`, cron `0 */3 * * *` + `workflow_dispatch` with a `--dry-run` input): the authoritative backstop for names written before the filter shipped, clever evasions, or direct-to-Firebase writes. Uses the **union** of two detectors — `obscenity` (boundary- + whitelist-aware matching of the **fuller** `HATE_SPEECH_TERMS` list, so colliding terms like coon/spic/paki are caught WITHOUT flagging raccoon/Nigeria/Pakistan; whitelist enumerated in the script) and the client normalizer (aggressive leetspeak obscenity's transformer misses, e.g. `tr4nny→tranny`). Scrubs offending `daily/{date}/{pushId}.name` and `weekly/{weekStart}/{uid}.name` to **"Anonymous"** (replace, not delete — the score/streak isn't the problem, the name is). The user can set a clean name anytime.
- **Term lists live in `src/logic/hateSpeechTerms.js`** — a deliberately separate, content-warned blocklist module. Two exports: `HATE_SPEECH_TERMS_CLIENT_SAFE` (low-collision, for naive client substring) and `HATE_SPEECH_TERMS` (fuller, server-only, safe because obscenity is boundary-aware). NEVER feed the fuller list to a naive substring matcher.
